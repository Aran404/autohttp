# autohttp — Generated Script Runtime

Date: 2026-06-21

## Generated Script Runtime

Generated scripts must be standalone. They cannot require `autohttp`, the Python AI worker, gRPC services, or `g4f`.

### Runtime Modes

Generated output supports two execution modes.

**Pure HTTP Mode** is default and preferred.

Includes only:

- HTTP client
- Cookie jar
- Header/body templating
- JSON/form/HTML extractors
- Response validation
- Dynamic value binding
- Optional retry/backoff
- Optional proxy support

This should cover most functional replay cases.

**Browser-Assisted Mode** is used only when the graph proves pure HTTP cannot reproduce a value.

Examples:

- Token generated by WebCrypto
- Token depends on DOM/canvas/browser fingerprint
- Challenge requires real browser execution
- Captcha/challenge flow cannot be represented as HTTP-only
- Required request is initiated by opaque frontend code

Browser-assisted mode may include hooks to launch CamoFox or another configured browser, but that dependency belongs to the generated script, not to `autohttp`.

### Runtime Inclusion Rule

Generated scripts should include only what they need.

- If a flow only needs HTTP and JSON extraction, do not include JavaScript or captcha code.
- If a flow needs HTML hidden-input extraction, include an HTML extractor.
- If a flow needs captcha solving, include the captcha provider interface and configured provider adapter.
- If a flow needs browser fallback, include the minimal CamoFox launch/call logic.

### Go Runtime

Go output should prioritize speed and concurrency.

Core pieces:

- `http.Client`
- Cookie jar
- Context-aware requests
- Concurrent-safe state store
- Extraction functions
- Request builders
- Operation functions
- Optional TLS/fingerprint-aware client later
- Optional JavaScript runtime only when required

Generated Go should expose logical operations when possible:

```go
client := NewClient(Config{ProxyURL: proxy})
session, err := client.Login(ctx, username, password)
err = client.SubmitTargetAction(ctx, session, input)
```

### Python Runtime

Python output should be simpler and ergonomic.

Core pieces:

- `httpx` or a stdlib-compatible client choice
- Cookie jar
- Extraction functions
- Request builders
- Operation methods
- Optional async mode later

Python output is useful for users who want readability and easy modification.

### State Handling

Runtime state is explicit.

State can include:

- Cookies
- Extracted tokens
- Storage-like values
- Captcha solutions
- User inputs
- Operation results

No dynamic state should be hardcoded if the analyzer marked it as volatile.

### Failure Handling

Generated scripts should fail clearly.

Examples:

- Missing expected response field
- Unexpected status code
- Token extractor matched nothing
- Captcha provider failed
- Browser-assisted node unavailable
- Final success condition not met

The generated runtime should report the failed graph node or operation name.

### Non-Goals For v1

Do not make generated scripts self-healing through AI. There is no LLM dependency in generated scripts. If a site changes, the user re-records or edits the generated code.
